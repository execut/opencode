# Модель чтения (read-model)

Обязательные опции: ddd-light

Модель чтения используется, когда необходимо отделить бизнес-модель операций чтения от бизнес-модели операций записи.

Опция применяется всегда совместно с `ddd-light`, потому что модель чтения является вспомогательной моделью вокруг домена, а не заменой доменного слоя.

Модель чтения нужна, когда:

* чтение требует отдельной бизнес-логики;
* для чтения нужны DTO, отличающиеся от агрегата;
* нельзя или неудобно отдавать наружу состояние агрегата напрямую;
* нужно оптимизировать сложные выборки;
* нужно собрать данные из нескольких таблиц или агрегатов;
* нужно построить read-side модель для API, CLI, отчётов или истории.

Модель чтения не должна менять состояние доменного агрегата.

## Правила реализации

* Модель чтения должна находиться отдельно от доменного агрегата.
* Модель чтения не является Aggregate Root.
* Модель чтения не должна использоваться для выполнения команд.
* Команды должны работать через доменный агрегат и его репозиторий.
* Запросы на чтение должны работать через отдельный read-repository или query-handler.
* Read-model может иметь собственные DTO, фильтры, сортировки, пагинацию и структуры результата.
* Read-model может читать данные напрямую из таблиц инфраструктуры, если это не нарушает бизнес-правила записи.
* Read-model может быть построена по событиям, если используется `business-events`.
* Read-model не должна содержать инварианты записи. Инварианты записи должны оставаться в domain.
* Read-model может содержать правила представления и правила чтения: например, какие статусы видны пользователю, как собирается история, какие поля скрываются.
* Если для чтения нужна бизнес-валидация входных параметров, её можно оформить через value objects.

Пример структуры без `application-layer`:

```text
/
  domain/
    product.go
    repository.go
    product_test.go
  readmodel/
    product.go
    repository.go
    query.go
  service.go
  repository.go
  read_repository.go
```

Пример структуры совместно с `application-layer`:

```text
/
  domain/
    product.go
    repository.go
    product_test.go
  readmodel/
    product.go
    repository.go
  application/
    commands/
      create_product.go
      rename_product.go
    queries/
      get_product.go
      list_products.go
      get_product_history.go
  repository.go
  read_repository.go
```

## В комбинации с другими опциями

### business-events

Read-model может строиться или обновляться на основе бизнес-событий.

В этом случае события агрегата становятся источником изменений для модели чтения.

Например:

* `ProductCreated` создаёт запись в read-model;
* `ProductRenamed` обновляет название;
* `ProductApproved` меняет статус;
* `ProductRejected` меняет статус и причину отклонения;
* события также могут использоваться для построения истории.

Read-model не должна сама создавать бизнес-события записи. Она только использует уже возникшие события.

### Подписчики и диспетчеры событий

Read-model хорошо совмещается с подписчиками на события.

Подписчик может реагировать на события агрегата и обновлять read-side представление.

Пример:

```text
domain/
  event.go
readmodel/
  product.go
  projector.go
application/
  commands/
    rename_product.go
```

`projector.go` может быть подписчиком, который принимает событие и обновляет read-model.

Важно: если проектор обновляет read-model синхронно в рамках той же операции, ошибка проектора должна обрабатываться как часть выполнения команды. Если read-model обновляется асинхронно, нужно учитывать eventual consistency.

### application-layer

При наличии `application-layer` операции чтения должны быть вынесены в query-handlers.

Команды находятся в `application/commands`, запросы — в `application/queries`.

Командные обработчики работают с domain-агрегатами. Query-handlers работают с read-model.

### pseudo-cqrs

`read-model` является обязательной частью `pseudo-cqrs`.

В `pseudo-cqrs` модель чтения отделена на уровне кода, но может использовать ту же базу данных, те же таблицы и синхронное чтение.

### cqrs

В полноценном `cqrs` read-model обычно становится отдельной проекцией.

Она может обновляться через события, outbox, подписчиков или отдельные фоновые обработчики.

В этом случае read-model может быть eventually consistent относительно write-model.

### value-objects

Если параметры чтения имеют бизнес-смысл, они могут быть оформлены как value objects.

Например:

* `ProductID`;
* `User`;
* `Moderator`;
* `Page`;
* `Limit`;
* `Period`.

Но не нужно превращать все технические параметры чтения в VO без необходимости.

### optimistic-locking

Read-model может отдавать наружу версию агрегата, если клиенту нужно выполнять последующие команды с проверкой версии.

Но сама read-model не должна управлять optimistic locking. Проверка версии должна выполняться в write-side репозитории.

# Псевдо-CQRS (pseudo-cqrs)

Обязательные опции: application-layer + read-model

`pseudo-cqrs` — это облегчённый вариант CQRS, при котором команды и запросы разделены на уровне кода, но не обязательно разделены физически.

Эта опция используется, когда нужно явно отделить операции записи от операций чтения, но полноценная асинхронная CQRS-инфраструктура ещё не нужна.

`pseudo-cqrs` полезен, когда:

* команды уже имеют доменную модель;
* запросы требуют отдельной модели чтения;
* чтение не должно зависеть от структуры агрегата;
* нужно сделать код понятнее без брокеров, inbox, outbox и eventual consistency;
* нужно подготовить проект к полноценному CQRS в будущем.

## Правила реализации

* Команды и запросы должны быть разделены в application layer.
* Команды должны находиться в `application/commands`.
* Запросы должны находиться в `application/queries`.
* Команды должны работать через domain-агрегаты.
* Запросы должны работать через read-model.
* Команды не должны возвращать read-model.
* Запросы не должны менять состояние.
* Query-handlers не должны вызывать методы агрегата, меняющие состояние.
* Command-handlers не должны строить сложные read DTO.
* Read-model может читать данные из той же базы данных, что и write-model.
* Read-model может быть обычным SQL-view, отдельным query repository или отдельной таблицей.
* В `pseudo-cqrs` не обязательно использовать асинхронные события.
* В `pseudo-cqrs` не обязательно использовать отдельные базы данных.
* В `pseudo-cqrs` не обязательно использовать command bus.
* В `pseudo-cqrs` не обязательно использовать inbox.

Пример структуры:

```text
/
  domain/
    product.go
    repository.go
  readmodel/
    product.go
    repository.go
  application/
    application.go
    commands/
      create_product.go
      rename_product.go
      approve_product.go
      reject_product.go
    queries/
      get_product.go
      list_products.go
      get_product_history.go
  infrastructure/
    product_repository.go
    product_read_repository.go
  presentation/
    http/
      product_handler.go
```

## В комбинации с другими опциями

### presentation-layer

Presentation должен явно разделять внешние операции записи и чтения.

Но это разделение не должно приводить к тому, что presentation начинает реализовывать бизнес-логику.

### business-events

Business-events можно использовать для сохранения истории и построения read-model.

В `pseudo-cqrs` события могут обрабатываться синхронно в рамках одной операции.

### Подписчики и диспетчеры событий

Подписчики могут обновлять read-model после выполнения команды.

Это удобно, если read-model строится по событиям, но ещё не вынесена в отдельный асинхронный процесс.

### value-objects

Команды должны использовать value objects для входных данных, имеющих бизнес-смысл.

Запросы могут использовать value objects для параметров чтения, если эти параметры имеют бизнес-валидацию.

### optimistic-locking

Команды могут использовать optimistic locking.

Запросы могут возвращать текущую версию агрегата, чтобы клиент мог передать её в следующей команде.

### cqrs

`pseudo-cqrs` является обязательной базой для полноценного `cqrs`.

Переход к `cqrs` происходит, когда появляются:

* асинхронная обработка команд;
* command inbox;
* отдельные проекции чтения;
* eventual consistency;
* отдельные обработчики событий;
* более строгие гарантии идемпотентности.

# Архитектура Domain driven design

Domain driven design - это подход к разработке, основанный на моделировании реальных бизнес-процессов и их изолировании в слое domain.
Синонимы: предметно-ориентрированная разработка, DDD, бизнес-ориентированная разработка.
Этот скилл даёт практическую структуру папок, объясняет слои и термины, содержит рабочие рецепты для повседневных задач.

## Когда применять
- При старте нового сервиса, где бизнес-правила сложнее CRUD и важна модель предметной области.
- При переносе или реорганизации существующего кода в DDD-структуру.
- При рефакторинге проекта на основе этого подхода

## Возможные опции
Архитектуру в зависимости от задач можно усложнять различными опциями. Такие опции перечислены в подразделах.
Каждые опции совместимы с другими опциями. Про это упоминается в подразделах.

В проекте https://github.com/execut/architecture-bricks/ есть папка с кодом pkg.
Внутри неё есть подпапки, совпадающие по названию с опциями архитектуры.
Там находится общий код для функционирования опций. Папки отражают вложенность опций: каждый уровень добавляет опцию. Например:
- `pkg`
  - `optimistic-locking` - опция оптимистичных блокировок
    - `busines-events` - опция оптимистичных блокировок в комбинации с бизнес-событиями
      - `value-objects` - опция оптимистичных блокировок в комбинации с бизнес-событиями и объектами-значениями

- Комбинации приоритетней, чем основные варианты опций

Ещё в этом проекте есть в папке app примеры реализации одного и того же приложения с разными опциями. В ней пожно подсмотреть примеры применения.

### DDD light (ddd-light)
Совместимость с другими опциями: domain-events

DDD light - самый облегчённый вариант архитектуры на базе подхода DDD (Domain driven design).

Есть только слой domain без бизнес-событий. Всё остальное находится в прикладном слое и никак не поделено.

#### Структура папок

- `<context>/` - папка контекста или реализации приложения
  - `domain/` — папка для доменного (domain) слоя
    - `<aggregate>.go` - агрегат, основная сущность
    - `<aggregate>_test.go` - тесты для агрегата. Обязательно должен быть live-тест агрегата (подробнее в скиле tests)
  - `service.go` - прикладной слой, остальные слои размазаны тут
  - `repository.go` - релизация репозитория из domain-слоя

### Бизнес-события (business-events)
Обязательные опции: ddd-light

Во время действий над собой, агрегат вместо прямой модификации своих свойств, меняет их через события.
Заодно репозиторий может получить необходимые изменения через возникшие события, а не чтение свойств агрегата напрямую.

Появляются следующие файлы и папки:
- `<context>/` — папка ограниченного контекста или приложения, где <context> - название контекста или приложения
  - `domain/`
    - `event.go` - список возникающих бизнес-событий

- Для упрощения тест live-цикла не должен тестировать события и их методы. Тест должен проверять только изменения в состоянии сущности.
  - Каждый подтест цикла должен тестировать конкретное действие. Различные особые случаи с ошибками должны тестироваться вне live-цикла.
- В конструкторе сущности должен быть только идентификатор и внешние зависимости. Создание сущности должно быть только через её метод Create и порождение соответствующего события.
- Агрегаты и сущности теперь должны реализовывать интерфейс Entity
- Репозиторий теперь имеет такой интерфейс:
```go
interface Repository {
    Load(ctx context.Context, entity *Entity) error // репозиторий добывает и применяет все события через AddAndApplyEvent и затем вызывает CleanEventList
    Save(ctx context.Context, entity *Entity) error // чистит события через CleanEventList и сохраняет текущее состояние сущности в БД. При необходимости пишет из них историю.
    // другие вспомогательные методы
} 
```
- Тип событий определяется через switch event.Payload().(type)
- События нельзя создавать через отдельные конструкторы. Нужно использовать только конструктор NewEvent

### Подписчики и диспетчеры событий
Обязательные опции: (ddd-light + busines-events)

Опция полезна, когда необходимо реагировать на бизнес-события для целей вспомогательных поддоменов.

- `ddd`
  - `dispatcher.go` - диспетчер событий:
```go
type Subscriber interface {
    Handle(ctx context.Context, entity Entity, event Event) error
}

type Dispatcher struct {
    subscriberList []Subscriber
}

func NewDispatcher(subscriberList []Subscriber) *Dispatcher {
    return &Dispatcher{subscriberList: subscriberList}
}

func (d *Dispatcher) Dispatch(ctx context.Context, entity Entity) error {
    for _, subscriber := range d.subscriberList {
        for _, event := range entity.EventList() {
            err := subscriber.Handle(ctx, entity, event)
            if err != nil {
                return err
            }
        }
    }

    return nil
}
```

### Объекты-значения (value-objects)

Обязательные опции: нет

Обеспечивает согласованность данных в рамках специальных объектов-значений. Теперь в домен будут попадать только проверенные значения,
поскольку конструкторы объектов-значений пробрасывают ошибки при попытке создать объект в невалидном состоянии.
- Теперь все аргументы конструкторов и методов сущностей должны оборачиваться в объекты-значения.
- Вся валидация из других слоёв должна переехать на объекты-значения.
- Все значения бизнес-событий тоже должны быть объектами-значениями

### Оптимистичная блокировка (optimistic-locking)

Обязательные опции: нет

Предназначена для исключения ситуации конкурентной записи. Когда агрегат читается из БД, меняется в параллельном потоке и его изменения перезаписываются в текущем.
В таблицу необходимо добавить поле version. По-умолчанию оно 0.
При обновлении записей необходимо делать инкремент значения версии прямо в запросе: `UPDATE ... SET version = version + 1, остальные поля WHERE version=$version`.
Где $version - это версия, полученная при чтении агрегата.
В случае, если после обновления не поменялось ни одной записи, нужно пробрасывать ошибку, что запись уже поменялась другим запросом.

#### В комбинации с другими опциями
##### value-objects
Версия заворачивается в VO Version, создать который невозможно при отрицательной версии.

##### ddd-light
У агрегата появляется новое поле с версией.
- При вызове AddAndApplyEvent в сущности, необходимо передавать следующую версию и при UPDATE делать минус 1 от неё. Например:
`UPDATE ... SET version = version + 1, остальные поля WHERE version=$version`, где $version - это версия из события минус 1.

##### business-events
События теперь становятся структурами, которые принимают на входе версию. Например:

##### business-events+value-objects
В комбинации с объектами-значениями, вместо int64 появляется VO Version.

### Состояния (states)
Обязательные опции: ddd-light

Все свойства сущностей переезжают в специальный DTO `<Entity>State`, где `<Entity>` - название сущности.
- Теперь, чтобы получить значение свойства сущности, её геттер обращается к State и пробрасывает его свойство.
- Ещё у сущности появляется метод State() <Entity>State. С его помощью репозиторий может доставать 

#### business-events
Теперь в событиях есть свойство с интерфейсом State interface{}. Оно передаётся в конструктор NewEvent.

### DDD high

DDD high - это самый навороченный вариант архитектуры на базе подхода DDD

Есть слои:
- Contract
- Application
- Domain
- Infrastructure
- Presentation

#### Структура папок
Простая и проверенная структура. Все слои конкретного bounded context помещаем в папку контекста:

- `<context>/` — папка ограниченного контекста, где <context> - название контекста (в репозитории-примере это promotions). Контексты обычно лежат в папке service
  - `domain/` — папка для доменного (domain) слоя
    - `events.go` — доменные события и их структуры.
  - `application/` — оркестрация: use-cases, обработчики команд/процессоров, контракты
    - `contract/` - интерфейсы, контракт приложения
      - `application.go` - находится основной интерфейс приложения Application. В него тут встраиваются три других: Commands, Processors, Queries.
      - `commands.go` - интерфейс для команд (операций записи) контекста
      - `processors.go` - интерфейс для процессоров (фоновых операций), нескольких команд
      - `queries.go` - интерфейс для запросов
    - `queries/` - реализация операций чтения. Тут находятся обработчики (хандлеры), которые реализуют интерфейсы запросов из контракта 
    - `commands/` - реализация команд. Тут находятся обработчики (хандлеры), которые реализуют интерфейсы запросов из контракта
    - `processors/` - реализация процессоров из контракта
  - `infrastructure` — адаптеры: реализации репозиториев, внешние клиенты, БД и API-клиенты
  - `presentation` — API / CLI / handlers — слой взаимодействия с внешним миром
- `pkg` - пакет для обобщённого кода, который можно переиспользовать в разных контекстах
  - `ddd` - общий код для подхода ddd, тоже разделяется по слоям:
    - `domain` - общий код для бизнес-логики
    - `infrastructure` - общий код для инфраструктуры

## Слои и их ответственность

### Contract

Слой содержит контракты контекста. Ничего не знает про другие слои. Содержит интерфейсы, через которые клиенты общаются с контекстом.

### Application
Прикладной слой. Реализует контракты из предыдущего слоя.

- Знает только про domain-слой
- Оркеструет операции домена: загружает агрегаты через репозитории, вызывает методы агрегатов, сохраняет изменения.
- Определяет контракты команд/сообщений (DTO для входа в приложение).
- Может содержать транзакционную логику, но не бизнес-инварианты.

#### Обработчики
Внутри прикладного слоя есть обработчики нескольких видов: команды, запросы и процессоры. Каждый обработчик реализует
один метод приложения из контракта. Прикладной слой реализуется через встраивание в него обработчиков (хандлеров) нескольких видов.
Каждый обработчик - это реализация одного метода из контракта.
Пример обработчика:
```go
type PromotionActualizeHandler struct {
//...
}

func NewPromotionActualizeHandler(...) PromotionActualizeHandler {
return PromotionActualizeHandler{repository: repository, promotionRepository: promotionRepository}
}

func (h *PromotionActualizeHandler) PromotionActualize(ctx context.Context, cmd contract.PromotionActualize) error {
//...
}
```

Хандлеры встраиваются в приложение через конструктор. Пример:
```go
func NewApplication(productGroupRepository domain.ProductGroupRepository, promotionRepository domain.PromotionRepository) contract.Application {
  return &App{
    //...
    appProcessors: appProcessors{
        processors.NewPromotionActualizeHandler(productGroupRepository, promotionRepository),
    },
  }
}
```

### Domain
**Синонимы**: domain-слой, предметный-слой, бизнес-слой

В этом слое находится модель предметной области, она моделирует реальные бизнес-процессы, их правила и логику.
Тут находится бизнес-модель и её объекты.

- Не зависит от других слоёв и частей приложения, кроме pkg и библиотек вроде lo
- Вся бизнес-валидация и инварианты находятся здесь.
- Все объекты в domain должны создаваться только через конструкторы для того, чтобы объект мог гарантировать свою целостность.
В случае ошибки, конструктор возвращает ошибку валидации
- В domain весь функционал необходимо обязательно покрывать unit-тестами.
- Минимизируйте побочные эффекты в domain: операции возвращают ошибки и генерируют события, но не выполняют I/O.
- Все ошибки должны быть объявлены в переменных с префиксом Err
- Слой ничего не знает про БД и про её реализацию. Не должно быть специальных конструкторов под БД.
- Все бизнес-типы должны жить в domain, а не импортироваться из внешних слоёв (contract/infrastructure).
- Если contract и domain используют одни и те же понятия, их нужно определить в domain и маппить в contract на границе.

#### Вспомогательный поддомен
Вспомогательный поддомен необходимо для того, чтобы моделировать бизнес-логику, строящуюся вокруг основного бизнес-слоя.
Примеры вспомогательного поддомена:
- Подписчик на бизнес-события для их интеграции в другой контекст
- Подписчик на бизнес-события для их ретрансляции в outbox или систему событий
- Бизнес-модель операции чтения

#### Сущности
**Сущность (Entity)** - это объект с идентичностью (например, Product, Promotion). Моделирует бизнес-объект из реального мира.
**Агрегат (Aggregate)** - группа сущностей и VO с корнем-агрегатом (Aggregate Root) — единственная точка входа для команд. Только через агрегаты команды меняют состояние контекста.
- Пишите мелкие, понятные методы в агрегате. Сложную крос-агрегатную логику выносите в Domain Service или Application.
- Ресиверы сущностей должны быть только указателями. Агрегаты запрещено передавать по значению. Везде используй указатели

#### Объекты-значения
**Объекты-значения (Value Object или VO)** - неизменяемый объект, сравниваемый по значению (Price, SKU). Моделирует существительное.

- VO являются неизменяемыми. Если возникает необходимость поменять объект, то это другой объект, его необходимо копировать.
- Копироваться VO должны только через специальные методы-фабрики внутри VO.
- Свойства VO должны быть публичными для упрощения доступа и кода, но эти свойства нельзя менять извне.
- Все ID тоже должны быть VO. Тем самым они инкапсулируют в себе свою реализацию
  - Если ID можно выразить как бизнес-объект и контекст это позволяет, то его нужно называть в виде существительного. Например: не UserID, а User.
  - Такие VO должны обеспечивать правильность значения ID в конструкторе. В таких случаях можно заносить библиотеки, которые отвечают за UUID

#### Бизнес-события
Бизнес-события (Domain Event) - это событие, описывающее факт в домене. С помощью событий аггрегаты перемещают своё состояние в инфраструктурный слой.
На бизнес-события могут подписываться подписчики из разных слоёв через паттерн издатель-подписчик.

#### Предметные сервисы
Предметные сервисы (Domain Service) - это бизнес-логика, которая не относится к сущности/агрегату и относящаяся к домену.

#### Repository
Интерфейс для загрузки/сохранения агрегата.

### Infrastructure
Инфраструктурный слой - это технический фундамент приложения, который обеспечивает связь доменной модели с внешним миром.

Он берет на себя все детали реализации, позволяя основному коду (домену) концентрироваться исключительно на бизнес-логике.
В инфраструктурном слое находятся:
- Реализация репозиториев, интерфейсы которых объявлены в домене
- Реализация интерфейсов для взаимодействия с внешними сервисами

- Знает только про слой domain

### Presentation
Слой представления - это реализация клиентов приложения. Тут находятся:
- Хандлеры для обработки внешних событий
- Консольные приложения

Слой presentation должен знать только про слой contract.

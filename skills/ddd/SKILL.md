# Архитектура Domain driven design

Domain driven design - это подход к разработке, основанный на моделировании реальных бизнес-процессов и их изолировании в слое domain.
Синонимы: предметно-ориентрированная разработка, DDD, бизнес-ориентированная разработка.
Этот скилл даёт практическую структуру папок, объясняет слои и термины, содержит рабочие рецепты для повседневных задач.

## Когда применять
- При старте нового сервиса, где бизнес-правила сложнее CRUD и важна модель предметной области.
- При переносе или реорганизации существующего кода в DDD-структуру.
- При рефакторинге проекта на основе этого подхода

## Возможные опции
Архитектуру в зависимости от задач можно усложнять различными опциями. Такие опции перечислены в подразделах.
Каждые опции совместимы с другими опциями. Про это упоминается в подразделах.

В проекте https://github.com/execut/architecture-bricks/ есть папка с кодом pkg.
Внутри неё есть подпапки, совпадающие по названию с опциями архитектуры.
Там находится общий код для функционирования опций. Папки отражают вложенность опций: каждый уровень добавляет опцию. Например:
- `pkg`
  - `optimistic-locking` - опция оптимистичных блокировок
    - `busines-events` - опция оптимистичных блокировок в комбинации с бизнес-событиями
      - `value-objects` - опция оптимистичных блокировок в комбинации с бизнес-событиями и объектами-значениями

- Комбинации приоритетней, чем основные варианты опций

Ещё в этом проекте есть в папке app примеры реализации одного и того же приложения с разными опциями. В ней пожно подсмотреть примеры применения.

# Архитектура Domain driven design

Domain driven design - это подход к разработке, основанный на моделировании реальных бизнес-процессов и их изолировании в слое domain.
Синонимы: предметно-ориентрированная разработка, DDD, бизнес-ориентированная разработка.
Этот скилл даёт практическую структуру папок, объясняет слои и термины, содержит рабочие рецепты для повседневных задач.

## Когда применять
- При старте нового сервиса, где бизнес-правила сложнее CRUD и важна модель предметной области.
- При переносе или реорганизации существующего кода в DDD-структуру.
- При рефакторинге проекта на основе этого подхода

## Структура папок
Простая и проверенная структура. Все слои конкретного bounded context помещаем в папку контекста:

- `service/<context>/` — папка ограниченного контекста, где <context> - название контекста (в репозитории-примере это promotions)
  - `domain/` — папка для доменного (domain) слоя
  - `application/` — оркестрация: use-cases, обработчики команд/процессоров, контракты
    - `contract/` - интерфейсы, контракт приложения
      - `application.go` - находится основной интерфейс приложения Application. В него тут встраиваются три других: Commands, Processors, Queries.
      - `commands.go` - интерфейс для команд (операций записи) контекста
      - `processors.go` - интерфейс для процессоров (фоновых операций), нескольких команд
      - `queries.go` - интерфейс для запросов
    - `queries/` - реализация операций чтения. Тут находятся обработчики (хандлеры), которые реализуют интерфейсы запросов из контракта 
    - `commands/` - реализация команд. Тут находятся обработчики (хандлеры), которые реализуют интерфейсы запросов из контракта
    - `processors/` - реализация процессоров из контракта
  - `infrastructure` — адаптеры: реализации репозиториев, внешние клиенты, БД и API-клиенты
  - `presentation` — API / CLI / handlers — слой взаимодействия с внешним миром
- `pkg` - пакет для обобщённого кода, который можно переиспользовать в разных контекстах
  - `ddd` - общий код для подхода ddd, тоже разделяется по слоям:
    - `domain` - общий код для бизнес-логики
    - `infrastructure` - общий код для инфраструктуры

## Слои и их ответственность

### Contract

Слой содержит контракты контекста. Ничего не знает про другие слои. Содержит интерфейсы, через которые клиенты общаются с контекстом.

### Application
Прикладной слой. Реализует контракты из предыдущего слоя.

- Знает только про domain-слой
- Оркеструет операции домена: загружает агрегаты через репозитории, вызывает методы агрегатов, сохраняет изменения.
- Определяет контракты команд/сообщений (DTO для входа в приложение).
- Может содержать транзакционную логику, но не бизнес-инварианты.

#### Обработчики
Внутри прикладного слоя есть обработчики нескольких видов: команды, запросы и процессоры. Каждый обработчик реализует
один метод приложения из контракта. Прикладной слой реализуется через встраивание в него обработчиков (хандлеров) нескольких видов.
Каждый обработчик - это реализация одного метода из контракта.
Пример обработчика:
```go
type PromotionActualizeHandler struct {
//...
}

func NewPromotionActualizeHandler(...) PromotionActualizeHandler {
return PromotionActualizeHandler{repository: repository, promotionRepository: promotionRepository}
}

func (h *PromotionActualizeHandler) PromotionActualize(ctx context.Context, cmd contract.PromotionActualize) error {
//...
}
```

Хандлеры встраиваются в приложение через конструктор. Пример:
```go
func NewApplication(productGroupRepository domain.ProductGroupRepository, promotionRepository domain.PromotionRepository) contract.Application {
  return &App{
    //...
    appProcessors: appProcessors{
        processors.NewPromotionActualizeHandler(productGroupRepository, promotionRepository),
    },
  }
}
```

### Domain
**Синонимы**: domain-слой, предметный-слой, бизнес-слой

В этом слое находится модель предметной области, она моделирует реальные бизнес-процессы, их правила и логику.
Тут находится бизнес-модель и её объекты.

- Не зависит от других слоёв и частей приложения, кроме pkg и библиотек вроде lo
- Вся бизнес-валидация и инварианты находятся здесь.
- Все объекты в domain должны создаваться только через конструкторы для того, чтобы объект мог гарантировать свою целостность.
В случае ошибки, конструктор возвращает ошибку валидации
- В events.go: находятся типы доменных событий и их структуры.
- В domain весь функционал необходимо обязательно покрывать unit-тестами.
- Минимизируйте побочные эффекты в domain: операции возвращают ошибки и генерируют события, но не выполняют I/O.
- Все ошибки должны быть объявлены в переменных с префиксом Err

#### Сущности
**Сущность (Entity)** - это объект с идентичностью (например, Product, Promotion). Моделирует бизнес-объект из реального мира.
**Агрегат (Aggregate)** - группа сущностей и VO с корнем-агрегатом (Aggregate Root) — единственная точка входа для команд. Только через агрегаты команды меняют состояние контекста.
- Пишите мелкие, понятные методы в агрегате. Сложную крос-агрегатную логику выносите в Domain Service или Application.

#### Value Object
**Объекты-значения (Value Object или VO)** - неизменяемый объект, сравниваемый по значению (Price, SKU).

- VO являются неизменяемыми. Если возникает необходимость поменять объект, то это другой объект, его необходимо копировать.
- Копироваться VO должны только через специальные методы-фабрики внутри VO.
- Свойства VO должны быть публичными для упрощения доступа и кода, но эти свойства нельзя менять извне.

#### Бизнес-события
Бизнес-события (Domain Event) - это событие, описывающее факт в домене. С помощью событий аггрегаты перемещают своё состояние в инфраструктурный слой.
На бизнес-события могут подписываться подписчики из разных слоёв через паттерн издатель-подписчик.

#### Предметные сервисы
Предметные сервисы (Domain Service) - это бизнес-логика, которая не относится к сущности/агрегату и относящаяся к домену.

#### Repository
Интерфейс для загрузки/сохранения агрегата.

### Infrastructure
Инфраструктурный слой - это технический фундамент приложения, который обеспечивает связь доменной модели с внешним миром.

Он берет на себя все детали реализации, позволяя основному коду (домену) концентрироваться исключительно на бизнес-логике.
В инфраструктурном слое находятся:
- Реализация репозиториев, интерфейсы которых объявлены в домене
- Реализация интерфейсов для взаимодействия с внешними сервисами

- Знает только про слой domain

### Presentation
Слой представления - это реализация клиентов приложения. Тут находятся:
- Хандлеры для обработки внешних событий
- Консольные приложения

Слой presentation должен знать только про слой contract.

## Событийная система
Во время команд, агрегат накапливает события в памяти. Application вызывает репозиторий.Save(ctx, aggregate) и репозиторий по событиям
обновляет информацию в БД.

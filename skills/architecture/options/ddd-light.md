# DDD light (ddd-light)
Обязательные опции: нет

DDD light - самый облегчённый вариант архитектуры на базе подхода DDD (Domain driven design).

Domain driven design - это подход к разработке, основанный на моделировании реальных бизнес-процессов и их изолировании в слое domain.
Синонимы: предметно-ориентированная разработка, DDD, бизнес-ориентированная разработка.

Есть только слой domain. Всё остальное находится в прикладном слое и никак не поделено.

## Структура папок

- `<context>/` - папка контекста или реализации приложения
  - `domain/` — папка для доменного (domain) слоя
    - `<aggregate>.go` - агрегат, основная сущность
      - Внутри находятся переменные для ошибок бизнес-инвариантов
    - `<aggregate>_test.go` - тесты для агрегата. Обязательно должен быть live-тест агрегата (подробнее в скиле tests)
    - `<entity>.go` - (опционально) вложенная в агрегат сущность с названием <entity>
  - `service.go` - прикладной и инфраструктурный слой

## Принципы

Любое приложение состоит из слоёв. При разных комбинациях опций, слои могут как соединяться вместе, так и разъединяться.
Они должны быть отдельны, если это указано в архитектурной опции явно.

### Contract

Слой содержит контракты контекста. Ничего не знает про другие слои. Содержит интерфейсы, через которые клиенты общаются с контекстом.

### Application
Прикладной слой. Реализует контракты из предыдущего слоя.

- Знает только про domain-слой
- Оркеструет операции домена: загружает агрегаты через репозитории, вызывает методы агрегатов, сохраняет изменения.
- Определяет контракты команд/сообщений (DTO для входа в приложение).
- Может содержать транзакционную логику, но не бизнес-инварианты.

### Domain
**Синонимы**: domain-слой, предметный-слой, бизнес-слой

В этом слое находится модель предметной области, она моделирует реальные бизнес-процессы, их правила и логику.
Тут находится бизнес-модель и её объекты.

- Не зависит от других слоёв и частей приложения, кроме pkg и библиотек вроде lo
- Вся бизнес-валидация и инварианты находятся здесь.
- Все объекты в domain должны создаваться только через конструкторы для того, чтобы объект мог гарантировать свою целостность.
  В случае ошибки, конструктор возвращает ошибку валидации
- В domain весь функционал необходимо обязательно покрывать unit-тестами.
- Минимизируйте побочные эффекты в domain: операции возвращают ошибки и генерируют события, но не выполняют I/O.
- Все ошибки должны быть объявлены в переменных с префиксом Err
- Слой ничего не знает про БД и про её реализацию. Не должно быть специальных конструкторов под БД.
- Все бизнес-типы должны жить в domain, а не импортироваться из внешних слоёв (contract/infrastructure).
- Если contract и domain используют одни и те же понятия, их нужно определить в domain и маппить в contract на границе.
  - `application` сам импортирует типы из `domain` и мапит их в `contract`.
- Запрещён импорт из других слоёв в domain

#### Вспомогательный поддомен
Вспомогательный поддомен необходим для того, чтобы моделировать бизнес-логику, строящуюся вокруг основного бизнес-слоя.
Примеры вспомогательного поддомена:
- Подписчик на бизнес-события для их интеграции в другой контекст
- Подписчик на бизнес-события для их ретрансляции в outbox или систему событий
- Бизнес-модель операции чтения

#### Сущности
**Сущность (Entity)** - это объект с идентичностью (например, Product, Promotion). Моделирует бизнес-объект из реального мира.
**Агрегат (Aggregate)** - группа сущностей и VO с корнем-агрегатом (Aggregate Root) — единственная точка входа для команд. Только через агрегаты команды меняют состояние контекста.
- Пишите мелкие, понятные методы в агрегате. Сложную кросс-агрегатную логику выносите в Domain Service или Application.
- receiver-ы сущностей должны быть только указателями. Сущности запрещено передавать по значению. Везде используй указатели
- Бизнес-инварианты агрегата должны проверяться внутри самого агрегата.
  Если действие пытается перевести агрегат в неконсистентное состояние, метод агрегата должен вернуть бизнес-ошибку из domain, а не булевый флаг.
  Application layer не должен самостоятельно интерпретировать состояние агрегата и выбирать бизнес-ошибку.
- Импорт contract в domain запрещён
- Бизнес-ошибки объявляются в domain, а не импортируются из contract.
- Все бизнес-ошибки должны мапиться из domain слоя в contract на прикладном уровне. Прикладной слой мапит domain.Err... в contract.Err....
- Прикладной слой выполняет только координацию (вызовы репозитория и методов агрегата). Вся бизнес-логика — строго в domain.
- Вся бизнес-логика должна быть реализована в методах доменного агрегата (или сущности). Прикладной слой (service.go) выполняет только координацию: вызывает методы репозитория и методы агрегата, но не содержит бизнес-логики (вычислений, проверок, принятия решений).

#### Объекты-значения
**Объекты-значения (Value Object или VO)** - неизменяемый объект, сравниваемый по значению (Price, SKU). Моделирует существительное.

- VO являются неизменяемыми. Если возникает необходимость поменять объект, то это другой объект, его необходимо копировать.
- Копироваться VO должны только через специальные методы-фабрики внутри VO.
- Свойства VO должны быть публичными для упрощения доступа и кода, но эти свойства нельзя менять извне.
- Все ID тоже должны быть VO. Тем самым они инкапсулируют в себе свою реализацию
  - Если ID можно выразить как бизнес-объект и контекст это позволяет, то его нужно называть в виде существительного. Например: не UserID, а User.
  - Такие VO должны обеспечивать правильность значения ID в конструкторе. В таких случаях можно заносить библиотеки, которые отвечают за UUID

#### Предметные сервисы
Предметные сервисы (Domain Service) - это бизнес-логика, которая не относится к сущности/агрегату и относящаяся к домену.

#### Repository
Интерфейс для загрузки/сохранения агрегата.

### Infrastructure
Инфраструктурный слой - это технический фундамент приложения, который обеспечивает связь доменной модели с внешним миром.

Он берет на себя все детали реализации, позволяя основному коду (домену) концентрироваться исключительно на бизнес-логике.
В инфраструктурном слое находятся:
- Реализация репозиториев, интерфейсы которых объявлены в домене
- Реализация интерфейсов для взаимодействия с внешними сервисами
- Знает только про слой domain

### Presentation
Слой представления - это реализация клиентов приложения. Тут находятся:
- Хандлеры для обработки внешних событий
- Консольные приложения

Слой presentation должен знать только про слой contract.

## В комбинации с другими опциями
### repository

В структуре появляются ещё файлы:

- `<context>/`
  - `domain/`
    - `repository.go` - интерфейс репозитория агрегата. Содержит в аргументах и возвращаемых значениях только структуры из domain.
    - `<entity>_repository.go` - (опционально) репозиторий сущности с названием <entity>
  - `repository.go` - реализация репозитория из интерфейса в domain
  - `<entity>_repository.go` - (опционально) реализация репозитория сущности с названием <entity>

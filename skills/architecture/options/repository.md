# Репозиторий (repository)

## Обязательные опции
нет

## Принципы
Репозиторий» (Repository) - это шаблон проектирования, который создаёт абстракцию между
бизнес‑логикой приложения и механизмом хранения данных.
- Вся работа с хранилищем/базой данных должна быть инкапсулирована в отдельной структуре-репозитории.  
- Прикладной сервис/сценарий не должен напрямую открывать соединения, выполнять SQL-запросы или знать детали хранения данных.
- Для каждого агрегата или сущности, сохраняемой в хранилище, должен быть выделен отдельный репозиторий.
- Репозиторий — это специальная структура, инкапсулирующая процесс работы с базой данных или другим хранилищем.
- Прикладной сервис не должен напрямую выполнять SQL-запросы, открывать соединения с базой или работать с транзакционными деталями хранения.
- Прикладной сервис должен обращаться к хранилищу только через методы репозитория.
- Репозиторий должен возвращать доменные или контрактные ошибки, а не раскрывать наружу ошибки конкретной инфраструктуры, если они имеют бизнес-смысл.

В данном шаблоне есть только один слой, где лежит код. Структура:
- `<context>/` - папка контекста или реализации приложения
    - `service.go` — реализация контекста или приложения, обращения к БД
    - `repository.go` - реализация репозитория основной модели
    - `*_repository.go` - реализация репозиториев других моделей
    - `*.go` - другие вспомогательные файлы

## Антипаттерны
Нельзя:
- `Service.CreateProduct` напрямую вызывает `pgx.Connect` и `conn.Exec`.
Нужно:
- `Service.CreateProduct` вызывает `productRepository.Create(ctx, product)`.
- `ProductRepository.Create` внутри себя выполняет SQL-запрос и преобразует ошибки БД.

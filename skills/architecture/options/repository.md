# Репозиторий (repository)

## Принципы
Репозиторий (Repository) - это шаблон проектирования, который создаёт абстракцию между
бизнес‑логикой приложения и механизмом хранения данных.
- Вся работа с хранилищем/базой данных должна быть инкапсулирована в отдельной структуре-репозитории.  
- Прикладной сервис/сценарий не должен напрямую открывать соединения, выполнять SQL-запросы или знать детали хранения данных.
- Для каждого агрегата или сущности, сохраняемой в хранилище, должен быть выделен отдельный репозиторий.
- Репозиторий — это специальная структура, инкапсулирующая процесс работы с базой данных или другим хранилищем.
- Прикладной сервис не должен напрямую выполнять SQL-запросы, открывать соединения с базой или работать с транзакционными деталями хранения.
- Прикладной сервис должен обращаться к хранилищу только через методы репозитория.
- Если в репозитории возникла инфраструктурная ошибка, если они имеют бизнес-смысл, необходимо их мапить в бизнес-ошибку. Например ErrProductNotFound, ErrProductAlreadyExists, 
    ErrProductAlreadyChanged. Ошибки бизнес-правил, например ErrProductAlreadyApproved, должны формироваться вне репозитория.
- Репозиторий не должен писать в свои свойства состояние данных из БД, он должен содержать только логику.
- Факт наличия записи в БД можно проверять попыткой её вставить. За счёт уникальных ключей в БД, БД вернёт ошибку
- Репозиторий не должен содержать бизнес-логику. Он отвечает только за сохранение, загрузку, удаление и техническое преобразование данных/ошибок хранилища. Все бизнес-решения,                                                
  проверки статусов, правила переходов состояния и создание бизнес-событий должны находиться в других слоях, а не в репозитории.

В данном шаблоне есть только один слой, где лежит код. Структура:
- `<context>/` - папка контекста или реализации приложения
    - `service.go` — реализация контекста или приложения, обращения к БД
    - `repository.go` - реализация репозитория основной модели
    - `*_repository.go` - реализация репозиториев других моделей
    - `*.go` - другие вспомогательные файлы

## Антипаттерны

### Работа с БД из сервиса
Нельзя:
- `Service.CreateProduct` напрямую вызывает `pgx.Connect` и `conn.Exec`.
Нужно:
- `Service.CreateProduct` вызывает `productRepository.Create(ctx, product)`.
- `ProductRepository.Create` внутри себя выполняет SQL-запрос и преобразует ошибки БД.

### Бизнес-логика
Нельзя:
- `ProductRepository.Approve` проверяет, что товар уже approved, решает вернуть `ErrProductAlreadyApproved` и создаёт событие одобрения.
Нужно:
- `Service.ApproveProduct` загружает товар через репозиторий, проверяет бизнес-правило, принимает решение и вызывает методы репозитория для сохранения изменений/событий.
- `ProductRepository` только выполняет SQL и маппит технические ошибки хранения.

## В комбинации с другими опциями
### ddd-light
- Репозиторий не должен импортировать contract и не должен возвращать ошибки уровня contract.
- Репозиторий возвращает только инфраструктурные ошибки или доменные ошибки из domain.
- Маппинг domain.Err... в contract.Err... выполняется только в application/service слое.

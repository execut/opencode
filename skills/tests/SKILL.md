# Основные правила

- Каждый тест должен придерживаться паттерна AAA.
  - Каждый тест должен состоять из трёх блоков, резделённых переносами:
    - Подготовка (arrange) - подготовка тестовых данных, моков и тестируемого объекта
    - Действие (act) - вызов тестируемого метода
    - Проверка (assert) - проверка
  - Пример теста по паттерну AAA (комментарии в тесте не нужно делать, они для пояснения):
```go
    t.Run("success", func(t *testing.T) {
        // подготовка
        t.Parallel()
        id := uuid.New()

        // действие
        product, err := domain.NewProduct(id, "Coffee")

        // проверка
        require.NoError(t, err)
        require.Equal(t, id.String(), product.ID())
        require.Equal(t, "Coffee", product.Name())
    })
```
- Результаты должны проверяться через библиотеку github.com/stretchr/testify
- Все подтесты должны называться согласно паттерну "when_ДЕЙСТВИЕ_then_ОПИСАНИЕ_РЕЗУЛЬТАТА"
- Все тесты должны быть параллельны за счёт конструкции `t.Parallel()`, кроме последовательных тестов жизненного цикла сущностей
- Тесты должны быть в отдельном package, названном согласно паттерну `<package>_test`, где `<package>` - название пакета тестируемого объекта
- Каждый файл тестов должен тестировать один объект. Название теста должно соответствовать паттерну `<object_file>_test.go`, где `<object_file>` - название файла тестируемого объекта
- Общий код по тестам нужно собирать в файле tests_common/common.go Тестовые данные необходимо собирать там в переменных с префиксом test.
- Тестовые конструкторы необходимо называть с префиксом NewTest, чтобы их отличить от других конструкторов.
- Для моков необходимо всегда использовать библиотеку go.uber.org/mock/mockgen
  - Каждый мок генерировать нужно только через директиву //go:generate в начале теста, где нужен мок.
  - Моки нужно складывать в папке tests_common/mocks/
- Каждый тест должен тестировать только один случай

# Тест жизненного цикла сущностей

Тест жизненного цикла сущности нужно писать, когда необходимо проверить один связный сценарий работы с одной сущностью через публичный контракт приложения: создание, чтение, изменение состояния, историю событий, удаление или другие последовательные этапы.

Такой тест подходит для интеграционных и контрактных проверок, где каждый следующий шаг опирается на состояние, созданное предыдущим шагом. Не нужно использовать lifecycle-тест для независимых проверок валидации, ошибок, граничных случаев или набора однотипных кейсов: такие сценарии должны оставаться отдельными параллельными тестами или table-driven тестами.

Правила написания lifecycle-теста:

- Подтесты нужно называть не через паттерн как в основных правилах. В названии в snake_case нужно писать кратко описание проверяемого этапа жизненного цикла сущности. Например: "create", "approve", "create_and_autoapprove"
- В рамках одного теста должна проверяться одна сущность или один агрегат.
- Приложение, контекст, идентификаторы сущности и общая подготовка создаются один раз в начале теста.
- Очистка данных выполняется один раз через `t.Cleanup()`.
- Шаги жизненного цикла оформляются через `t.Run()` и называются по общему паттерну `when_ДЕЙСТВИЕ_then_ОПИСАНИЕ_РЕЗУЛЬТАТА`.
- Внутри шагов жизненного цикла не нужно вызывать `t.Parallel()`, потому что шаги должны выполняться последовательно и переиспользовать состояние одной сущности.
- Родительский lifecycle-тест может вызывать `t.Parallel()`, если он полностью изолирован от других тестов: использует уникальные идентификаторы и не выполняет глобальную очистку общих таблиц или хранилищ.
- Каждый шаг должен сохранять структуру ACT: подготовка для конкретного шага, действие, проверка. Общую подготовку, уже выполненную в начале lifecycle-теста, не нужно дублировать в каждом шаге.
- Проверки выполняются через `github.com/stretchr/testify`.

# Unit-тесты

- Файл unit-теста должен лежать в той же папке, что и тестируемый объект
